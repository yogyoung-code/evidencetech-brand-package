## 1. Sponsor 三红点裁决（2026-05-07 锁定）

### 1.1 R2 · 大区独发 + 事后 review

**裁决**：大区**可以独立发布**对外内容（接受市场立场）；增加**事后 review 流程**作为合规兜底（响应法务关切）。

**机制设计建议**（待 Sponsor 在 §3 确认）：

| 维度 | 建议值 |
|---|---|
| Review 频次 | 每周 |
| Review 范围 | 100% 大区独发内容（前期建立基线后可降至抽样）|
| Review 时间窗 | 发布后 7 天内由总部 + 法务完成 review |
| Review 责任人 | 市场总部 + 法务 |
| 违规处理 | 撤回（按 L2 收紧版时间窗）+ 公告勘误 + 大区责任人记录 |
| 累计违规 | 3 次违规取消大区独发权 / 收回总部审 |
| 例外 | 危机响应（撤稿 / 监管 / 自家 bug）仍走总部 + 法务双签前置 |

**对源文档的连锁修订**：

- 03e §6.1 RACI —— A/B 类拆分方案废弃；改为"大区独发 + 事后 review"统一规则 + 危机响应例外行
- 03e §6.2 本地化申请单流程 —— 废弃；改为大区直接发布 + 事后报备
- 03f §3 撤稿响应 —— 大区独发内容仍进实证库（继承 R3）；反向追踪覆盖大区发布物料
- 03a §5 危机手册 —— 大区独发内容触发危机时仍走原流程

---

### 1.2 R3 · SourceID + 工具配合

**裁决**：保留完整 SourceID 体系（接受法务立场，不"轻量化"）；通过**工具化**降低市场操作负担（响应市场关切）。

**工具能力清单**（建议作为 03f §6.1 的工具选型 RFC 起草要点）：

- 一键注册（草稿 → 提交 → 审批工作流）
- 自动反向追踪（一键查看哪些 claim 引用此 SourceID）
- 自动复审提醒（按 §2.4 时效配置季度 / 年度）
- 撤稿监测自动化（PubMed RSS / Retraction Watch RSS 自动入库 + 自动 @ 责任人）
- 文案 → SourceID 自动校验（发布前一键校验，缺漏标红 + 阻塞发布）
- 多角色协作 UI（市场起草 / 法务审 / 学术审 / 产品引用 / 管理员维护）
- Audit Trail 全留痕（不允许静默删除）

**工具选型**（双方兼容已结论）：**开源自建**——候选 NocoDB / Baserow / 自建 PG+FastAPI；v0.2 落地后由 Sponsor + 法务 + 工程三方决议选定。

**对源文档的修订**：

- 03f §2.1 SourceID 字段 —— 保留完整版（不分 L 级简化字段）；新增"工具化降低注册成本"承诺段落
- 03f §6.1 实证库工具选型 —— 更新为"开源自建" + 工具能力清单 + v0.2 落地后三方决议会议
