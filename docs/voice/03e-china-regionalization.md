## 2. 文案分层规则

### 2.1 术语密度

| 层级 | 术语策略 | 示例 |
|---|---|---|
| T1 / T2 / MNC | 学术英文保留，首次拼写全称 | PICO (Population, Intervention, Comparison, Outcome) |
| T3 / T4-A / 国内大型 Pharma | 中文先行，英文括注；二次提及可单用中文或缩写 | 研究者发起的临床研究 (IIT) |
| T4-B / Biotech / CRO / AI 公司 | 业务结果优先，术语密度最低 | 把"AI 检索优化 (AIO)"改为"让你的产品在医生 AI 搜索里被准确召回" |

**铁律**：03 §6 禁词清单（中英）在所有层级**无差别生效**。降低术语密度 ≠ 降低严谨性。

### 2.2 引用密度与类型

| 层级 | 首选引用源 | 备用引用 |
|---|---|---|
| T1 / T2 | SCI 文献 [n]（PMID / DOI） | 国家级指南 / 共识 |
| T3 | UpToDate 摘要 + 中文指南 | SCI 作补充 |
| T4-A 公立基层 | 国家指南 / 临床路径 / 行业共识 | 中文综述 |
| T4-B 新业态 | 客户 case ID（带客户书面同意）+ 业务结果数据 | 行业白皮书 |
| MNC | SCI + 全球指南（NCCN / ESMO / ACC 等） | 国家指南 |
| 国内大型 Pharma | 国家指南 + SCI | 国际指南 |
| Biotech / CRO | RCT 设计参数 + 行业基准 | 案例 |

**继承 03 §7.1**：所有层级的数据 claim 必带年份与来源；分层只影响"用什么源"，不影响"是否要源"。

> **强化条款**（v0.2 修订 · 2026-05-07）：本表是引用源**类型偏好**的唯一规则。**最低实证级别**（合规底线）按 03f §1.2 矩阵在所有层级**无差别生效**——T4-A 县域物料引用源数量可少（按本表偏好），但每条仍需 SourceID + 满足 03f §1.2 临床效果 claim 的 L3+L2 等最低要求。**降低术语密度 ≠ 降低实证级别**。

### 2.3 CTA 措辞分层

| 层级 | CTA 风格 | 示例 |
|---|---|---|
| T1 / T2 | 克制；学术导向 | 「查看完整方法学」 / 「下载白皮书」 / 「申请演示」|
| T3 / T4-A | 价值前置 + 低门槛 | 「免费试用 14 天」 / 「报名医院联系会」 / 「申请落地评估」|
| T4-B | ROI / 速度直述 | 「30 分钟看 demo」 / 「一个月评估 ROI」 / 「联系业务对接」|
| MNC | 长期品牌资产视角 | 「Q3 学术合作议题」 / 「年度 KOL 项目对接」|
| 国内大型 Pharma | 销量 + 学术混合 | 「学术拜访效能评估」 / 「品牌内容 AIO 治理」|
| Biotech / CRO | 速度 + 试错 | 「跑一次 pulse」 / 「48h 试用反馈」（必须满足 03 §3.4 的 24h 措辞规则）|

**铁律**：03 §6 + 03a §7.3 的"催促 / FOMO"禁用 CTA 在所有层级生效（"立即抢购" / "限时" / "速速" 等不可用）。

### 2.4 数据 claim 的本地化

| 错误 | 正确 |
|---|---|
| 引用一线三甲数据推销给县域 | 用同层 case；如必需跨层引用，**必须**在文中标注层级差异（参 §5.4） |
| 笼统说"全国 X 家医院使用" | 拆分"T1 X 家 / T2 X 家 / T3 X 家 / T4-A X 家 / T4-B X 家"，避免被误解为单层垄断 |
| 用 DRG 全面铺开城市的支付改革数据推 DIP 城市 | 分别引用 DRG / DIP 试点的对应数据（参 §4.1）|
| MNC case 跨借给 Biotech 客户 | 严禁——客户书面同意范围内才允许，且需经法务审 |

### 2.5 时间表达

T4 客户决策周期与 T1 不同；不要用统一的"30 天上线"模板。

| 层级 | 上线 / 部署节奏的真实表达 |
|---|---|
| T1 / T2 | 2–4 周（IT 评审 + 信息科联调）|
| T3 | 4–6 周 |
| T4-A 公立基层 | 6–10 周（含培训 + 院端协调）|
| T4-B 新业态 | 1–2 周（业态轻）|
| MNC | 季度级（合规 + 全球 IT 审）|
| 国内大型 Pharma | 月度级 |
| Biotech / CRO | 双周 |

**禁忌**：用 T4-B / Biotech 的速度去包装给 T1 / MNC 客户——会被识破并损失信任。

## 4. 政策与监管的地域差异

### 4.1 DRG / DIP

- **DRG 全面铺开城市**：DE 的 DRG 智能导航功能可作为核心 CTA
- **仅 DIP 试点城市**：用 DIP 替代 DRG 措辞，避免"我们用错系统"印象——文案改写："基于 DIP 病种分值的智能编码助手"
- **未试点城市**：作为前瞻性能力储备展示，不作为核心 CTA

**铁律**：永远不在同一份本地化稿件中混用 DRG / DIP 表述——按目标城市选定一种。

### 4.2 医保支付方差异

省级医保支付细则差异大；不在国家级正文中引用具体省的支付细则；如必需则在"地区版"内容中独立呈现。各省内容须经市场总部 + 法务双签，不允许大区独立发布。

### 4.3 互联网医院执业范围

- 各省执业范围 + 处方权限差异
- DE 在互联网医院场景的输出**不能**跨省提供"处方建议"型表述——用"用药参考"或"循证查询"替代
- T5 互联网医院客户的产品落地必须有"省级合规适配"层（产品侧已规划，参 `05-rectification-checklist.md`）

### 4.4 数据出境

- T1 / MNC 客户对数据出境敏感
- 任何"跨境训练 / 跨境推理"的对外表述必须经法务审
- 默认表达："数据本地化部署 / 训练数据未出境 / 模型推理在境内 X 区域"
- ❌ 禁忌："我们的模型基于全球数据训练" / "海外算力加持"——即使技术上成立也不在中国大陆物料中出现

### 4.5 集采与一致性评价的语境

DI 在向药企 BD 时，"AIO 检索优化"在仿制药一致性评价 / 国家集采品种语境下的措辞要谨慎——不允许出现"帮你在医生检索中盖过原研药"这种暗示性表述。

## 5. 跨层 / 跨地域敏感场景

### 5.1 县域 / 基层内容禁忌（T4-A）

- ❌ 暗示 "T4-A 医生水平不足需 AI 弥补"
- ✅ 改写："AI 让基层触达 T1 同等的循证资源"或"把 T1 的循证规范带到县域"
- ❌ "AI 替代基层医生" / "县域医生终于不用愁"
- ✅ 案例选取：T4-A → T4-A case 为主；不强行用 T1 case 反推

### 5.2 一线三甲内容禁忌

- ❌ "AI 已经超过医生" / "AI 比专家更准"——与 PITL 战略相悖（继承 03 §3.3）
- ❌ 暴露 "T1 之外能力更弱"
- ❌ 学术合作绑定整家医院——T1 学术合作以 PI 个人选择为主
- ❌ 把单一 KOL 的支持包装为"医院级背书"

### 5.3 民营 / 互联网医院禁忌（T4-B）

- ❌ 混用 "医院级 / 三甲级" 措辞做对比包装（T4-B 业态与 T1–T3 + T4-A 公立差异大）
- ❌ 引用三甲 case 推销给民营连锁
- ❌ 把医美 / 体检的"流量"案例与 DE 的临床决策严肃性混淆
- ❌ T4-B 互联网医院文案不能跨省提供"处方建议"型表述（继承 §4.3）；用"用药参考"或"循证查询"替代
- 互联网医院文案永远附"本输出不构成处方"声明（继承 03 §3.3 + §4.3）

### 5.4 跨层案例引用规则

- **同层优先**：T1 case 给 T1 客户，T4-A case 给 T4-A 客户，T4-B case 给 T4-B 客户，依此类推；**T4-A 与 T4-B 之间也算跨层**——业态差异大，不互通
- **跨层引用须标注**：在文中明确"本案例来自 T1 三甲，下沉到 T4-A 应用时 X / Y 部分有差异"
- **跨产业方引用须批准**：MNC case 不挪给 Biotech 客户；Biotech case 不放大为"行业普适"
- **不挪用兄弟业务**：梅斯医生社区平台的"下沉案例"不冒充梅斯 AI 直接交付（继承 03 §5.4）

### 5.5 区域热点 / 公共卫生事件

继承 03a §5.3：重大事件全平台暂停商业内容 24–72h；区域性重大事件（地震 / 局部疫情）按受影响区域**单独**暂停，不波及全国。

## 6. 团队动作与责任分工

### 6.1 内容生产 RACI（v0.2 修订 · 2026-05-07 · Sponsor R2 决议）

> **重大修订**：原表（按内容类型 7 行）废弃。改为统一"**大区独发 + 事后 review**"模型——大区获得对外发布权，总部 + 法务通过事后 review 兜底合规风险。**例外**：危机响应（撤稿 / 监管 / 自家 bug）仍走总部 + 法务**前置**双签（不允许大区独发触发危机响应）。

**统一规则**：

| 内容类型 | 大区权限 | 总部 + 法务参与方式 |
|---|---|---|
| **常规对外内容**（活动 / 团队 / 案例 / 政策评论 / 数据 claim 等所有日常内容）| **大区独立发布** | **事后 review**（按下表机制）|
| **危机响应**（撤稿 / 监管警示 / 自家产品 bug 媒体报道）| **不允许大区独发** | **前置双签**（按 03a §8.1 + 03f §3）|
| **客户共发布稿件** | 大区起草 | **前置双签**（客户法务 + 我方法务）|
| **MNC 学术合作 / 国内 Pharma 客户内容** | 大区起草 | **前置双签**（学术 + 法务 / KAM 协作）—— 涉客户合同条款，不适用大区独发 |
| **全国性核心 messaging** | 总部起草 | 市场负责人 + Sponsor 签 |

**事后 review 机制**：

| 维度 | 锁定值 |
|---|---|
| Review 频次 | 每周 |
| Review 范围 | 100% 大区独发内容（前期建立基线后可降至抽样）|
| Review 时间窗 | 发布后 7 天内由总部 + 法务完成 |
| Review 责任人 | 市场总部 + 法务 |
| 违规处理 | 撤回（按 03a §8.2 收紧时间窗）+ 公告勘误（按 03f §3.4 撤稿响应）+ 大区责任人记录 |
| 累计违规 | **3 次违规**取消大区独发权 / 收回总部前置审 |
| 抽查强度调整 | v0.2 落地后 6 个月评估违规率：< 5% → 转 50% 抽样 / 5–15% → 维持 100% / > 15% → 触发独发权回收 |

**铁律**：

- 大区独发**不豁免**任何 voice 层规则——03 主原则、本文档 §1–§5 的分层禁忌、03f SourceID 体系全部生效
- 大区独发的 case 引用仍**必须**进实证库（按 03f §1.2 + §2）；不进实证库的 case 视作违规
- 客户共发布稿件 / 危机响应**永远**走前置双签——不允许大区"先发后告"绕开
- 区域大会 / 巡回的现场材料按 03a §8.2 "排期与撤回"流程；**事后 review 不延伸覆盖现场口头讲话**——口头部分由大区主讲人个人合规承担

### 6.2 大区本地化操作（v0.2 修订）

**大区直接发布流程**：
