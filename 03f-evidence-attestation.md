## 0. 适用范围与边界

### 0.1 覆盖

所有对外的**声明（claim）**——含但不限于：

- 营销文案 / 白皮书 / 投资者材料 / 招股说明书 / 季报 / 年报

- **设计 / 视觉资产**——不属于"声明"

## 1. 声明类型与实证强度分级

### 1.1 实证强度（L1–L4）

| 级别 | 定义 | 例子 |
|---|---|---|
| **L1** | 公开来源 | 行业报告 / 公开媒体 / 公开行业白皮书 |
| **L2** | 可审计的内部数据 | 内部审计报告 / 产品后台日志 / 客户合同号关联的部署事实 |
| **L3** | 同行评议研究 | RCT / Meta 分析 / SCI / Cochrane / 国家级专家共识 |
| **L4** | 官方批文 / 书面合同 | NMPA/FDA 批文 / GVP 资质 / 客户书面同意书 / 学术 MOU / 监管复函 |

L1 < L2 < L3 < L4 是**实证强度**升序，不是覆盖范围或重要性。不同 claim 类型要求不同最低实证级别。

### 1.2 Claim 类型与最低实证要求

| Claim 类型 | 例子 | 最低实证 | 谁审 |
|---|---|---|---|
| 监管 / 合规 claim | "符合 GVP 标准" / "通过 X 等保测评" | **L4** 官方文件 | 法务 + 合规 |
| 临床效果 claim | "降低误诊率 X%" / "提升编码准确率到 Y%" | **L3** 同行评议研究 + L2 内部数据 | 产品 + 学术 + 法务 |
| 产品性能 claim | "Z 秒返回带源答复" / "覆盖 N 家三甲医院" | **L2** 内部审计（带 audit trail） | 产品 + 市场（涉客户名单时 + 法务） |
| 产品功能 claim | "提供 DRG 编码导航" / "支持纳米级引用" | **L2** 产品自有事实 | 产品 + 市场 |
| 市场规模 claim | "覆盖 X 万医生 / Y 家药企客户" | **L2** 内部审计 + 客户书面同意 | 市场 + 法务 |
| 行业评论 claim | "AI 在医学的 X 趋势" / 政策解读 | **L1** 公开来源 + 我方解读 | 市场 |
| 学术合作 claim | "与 X 教授 / Y 医院共同研究" | **L4** 书面 MOU / 合同 | 学术 + 法务 |
| 客户案例 claim | "Z 医院在 N 周内落地" | **L2** 部署事实 + **L4** 客户书面同意发布 | 市场 + KAM + 法务 |

**铁律**：L1 不能用于临床效果或监管 claim；L2 内部数据不能在缺少客户书面同意时点名客户；L3 文献结论不能直接转述为"我们的产品做到了"——必须区分"研究结论 vs 产品事实"。

## 2. 来源审批工作流（SourceID 体系）

### 2.1 SourceID 是什么

每一条引用源进入对外物料前必须先在**实证库**注册，获得唯一 SourceID（建议格式 `EVID-YYYY-NNNN`）。

> **工具化承诺**（v0.2 修订 · 2026-05-07 Sponsor 决议）：SourceID 体系的字段完整性不变，但通过**工具化**降低市场 / 学术 / 法务的注册操作负担——一键注册 / 自动反向追踪 / 自动复审提醒 / 撤稿监测自动化 / 文案 → SourceID 自动校验 / 多角色协作 UI / Audit Trail 全留痕。工具选型已锁定为**开源自建**（详见 §6.1）；v0.2 落地后由 Sponsor + 法务 + 工程三方决议会议选定具体方案，并在 30 天内启动工具选型 RFC。

**字段（强制）**：

- SourceID（唯一标识）
- 来源类型（SCI / Meta / RCT / 国家指南 / 行业共识 / 客户 case / 内部审计 / 监管文件 / 媒体 / 行业报告 / 等）
- 完整引用元数据（DOI / PMID / URL / 客户合同号 / 监管文号 / 内部审计 ID）
- 实证级别（L1–L4）
- **原始文件副本路径**（实证库本地存储，不可仅链接外网；外网链接可能失效）
- 录入日期 + 录入人
- 审批状态（pending / approved / revoked）
- 审批人（按 §1.2 表格路由）+ 审批日期 + 签名留痕
- 复审日期（按 §2.4 时效）
- **关联 claim 列表**（反向追踪——每条 claim 引用此源时回写）
- 备注（如客户同意书有效期、跨语言译稿署名、特殊声明义务）

### 2.2 注册流程

```
[内容生产者起草引用]
       ↓
[在实证库提交 SourceID 申请]
       ↓
[实证库管理员核验文件副本完整性]
       ↓
[按 §1.2 表格路由审批人]
       ↓
[审批人审 → approved / 驳回 / 补材料]
       ↓
[approved → SourceID 进入实证库可用列表]
       ↓
[发布前文案 claim → SourceID 自动校验]
```

**铁律**：

- 文案 claim 引用的 SourceID 必须状态为 **approved**——不允许引用 pending 或 revoked
- 一个 claim 可关联多个 SourceID（强证据链）
- SourceID 一经 approved 不允许静默修改；任何变更需走"版本号 + 修订记录"

### 2.3 不允许的来源（红线）

下列来源**永远不进入实证库**——即使内容方向正确：

- ❌ 维基百科 / 百度百科 / 知乎答主 / 自媒体作为唯一引用
- ❌ 撤稿文献（Retracted）——含已撤稿的 SCI / 已撤回的指南
- ❌ Predatory journal / 灰色出版商 / 非同行评议会议摘要的"独立证据"
- ❌ AI 生成的"文献综述"未经原文逐条核验——AI 工具仅作辅助检索，原文必读
- ❌ 客户口头同意的 case——必须书面同意书（含发布范围、有效期、撤回机制）
- ❌ 兄弟业务（梅斯医生社区）的非公开数据冒充梅斯 AI 直接交付
- ❌ 母品牌团队管辖范围内但未授权的母品牌资产数据
- ❌ 同行公司的内部材料（即使获取途径合法的公开演讲）
- ❌ 未公开发表的会议摘要（除非随后正式发表，且引用时注明发表状态）
- ❌ 截屏 / 二次引用——必须能定位到原始来源

### 2.4 来源时效性与复审

| 来源类型 | 推荐时效 | 强制复审节奏 |
|---|---|---|
| 临床指南 / 共识 | 5 年内或当前最新版 | 每年 |
| RCT / Meta 分析 | 不限（注明发表年份）| 每年 |
| 客户 case（带客户同意书） | 同意书有效期内 | **按客户合同期限或年度孰短**（确认仍在期内 + 客户未撤回）|
| 内部审计数据 | 1 年内 | 季度 |
| 监管文件 / 批文 | 当前有效 | 监管发布后即时 |
| 行业报告 / 公开数据 | 2 年内 | 半年 |
| 媒体引用 | 6 个月内 | 季度 |

**强制复审**：到期未复审的 SourceID 自动转 pending；任何依赖该 SourceID 的 claim 在新发布物料中不可用。

### 2.5 跨语言来源处理

- **引用英文文献**：保持英文原文 + DOI / PMID；中文物料中可附作者中文译稿，标注"作者翻译"——译稿不作为新源
- **引用中文文献**：保持中文原文 + 标准引用格式
- **跨语言转译禁忌**：不允许"译文 → 新 SourceID 注册"绕开原文核验
- 涉非英非中文献（日文 / 德文 / 法文）：附原文 + 经法务审核的中文译稿

## 3. 撤稿响应规程（完整版）

> 03a §5.1 是本节在社交场景的子集。本节是完整版——覆盖所有发布形态。

### 3.1 撤稿监测（Retraction Watch）

- **常规监测**：实证库管理员每月对全部 SourceID 跑一次 PubMed / Retraction Watch / 国家自然基金委 / NMPA 撤稿名单查询
- **被动入口**：审批人 / 客户 / 法务 / 媒体反馈也算入口；任何渠道发现立即报实证库
- **PubMed RSS 订阅**：建议自动化对所有 PMID 订阅"is_retracted" 状态变化（详见 §6.2）

### 3.2 T+0（发现当时）

- 实证库管理员在数据库标记该 SourceID 为 **revoked**
- 自动触发反向追踪：列出所有关联 claim 列表 + 这些 claim 出现的全部物料（社交 / 白皮书 / UI / 合同 / 内部）
- 内部群通报：市场 + 法务 + 学术 + 产品所有相关审批人（不在群内仅邮件不构成通报）
- **冻结**：所有受影响内容的新发布排期暂停

### 3.3 T+24h

- 影响范围评估表完成（哪些已发布 / 哪些在审 / 哪些是产品 UI / 哪些是合同附件）
- 评估表交市场负责人 + 法务双签
- 制定**分类处理计划**（按 §3.4）

### 3.4 T+24h 后的分类处理

| 物料类型 | 处理动作 |
|---|---|
| 社交媒体已发布 | 按 03a §5.1：**不删帖** + 在原帖加更新声明（红色 / 显著色块）+ 公众号发独立"勘误公告" |
| 白皮书 / 投资者材料 | 召回 PDF（实证库下架）+ 发新版（含撤稿事由说明）+ 通知已下载客户（如有名单则定向通知，无名单则官网公告）|
| 网站 / 落地页 | 立即更新；保留 archive 副本注明撤稿事由与时间 |
| **产品 UI 上的 claim** | 工程紧急 patch；如不能立刻修复（需走发版周期），**界面层加遮罩 / 占位符替换 + 说明文案**——不允许"边等修复边继续展示" |
| 客户合同附件 | 通知合同对应客户 + 提供修订版附件 + 法务出具说明 |
| 内部销售素材 | 群发召回通知 + 发新版（**不允许销售继续使用旧版**）|
| 招股 / 投资者关系材料 | 法务 + 投资者关系 + Sponsor 三签发"修订声明"；按 **HKEX Listing Rules**（重点：第 13.09 条 Inside Information / 第 14 章关联交易披露 / 第 17 章年度披露）+ **Inside Information Disclosure Rules（SFC）** 评估披露义务；如构成"具有重大性的信息"须立即披露；操作流程：法务 + 投资者关系 + Sponsor + 港交所联系人四方会签 |
| 媒体已转载 | 联系媒体要求更正；超过 **3 工作日**未更正发"勘误声明" |

### 3.5 复审与修订

- **T+1 周**：触发 PITL 网络对相关结论的复核——**结论本身**是否仍成立（一篇文献撤稿不一定推翻所有衍生结论；可能有其他源支撑）；如结论需修正发"修订公告"
- **T+2 周**：实证库归档撤稿事件（撤稿文献 ID + 撤稿原因 + 我方处理时间线 + 受影响 claim 修订记录）
- **T+1 个月**：触发**防御性复审**——同期 / 同 PI / 同主题的相关来源全部加跑一遍质量核查（避免下一个连环撤稿打个措手不及）

### 3.6 撤稿沉默期禁忌

- ❌ **删帖装作没事**——任何渠道（社交 / 网站 / PDF）的删而不告都是合规风险
- ❌ **私下改原文不留痕**——所有修订必须留痕（版本号 + 修订日期 + 事由）
- ❌ **推卸到"作者问题"逃避主动声明**——我们引用即承担引用责任
- ❌ **借机攻击同行**——撤稿是学术过程，不是营销机会
- ❌ **不通知客户 / 投资者 / 监管**而依靠"对方没注意"——通知义务按合同 / 上市规则执行

## 4. 数据 claim 的实证链条

每一个对外数据 claim 都必须有完整可追溯的链。

### 4.1 链条模板

```
[Claim 文本]
        ↓
[关联 SourceID 列表]（一个 claim 可有多个源）
        ↓
[原始文件副本]（实证库本地，不依赖外网）
        ↓
[审批 trail]（审批人 + 时间 + 签名留痕）
        ↓
[反向链接]（出现在哪些已发布物料；用于撤稿时的影响范围分析）
```

### 4.2 营销 / 投资者材料中的数据 claim

- 必须**每条**都有 SourceID
- 文末"参考文献"或"数据说明"段落给出完整列表（含 SourceID + 来源元数据）
- 上线 / 发布前对照实证库自动校验（建议工具化，详见 §6）
- 任何更改必须重新走审，不允许"小修"绕审
- 发布后 30 天内任何客户 / 媒体 / 监管对数据的质疑入实证库 ticket 系统跟踪

### 4.3 产品 UI 上的 claim

- UI 上的性能 / 准确率 / 覆盖范围 / 案例计数等数字**全部**有 SourceID
- 工程发版前对照"UI claim 清单"逐条核对（建议产品侧维护一份"UI claim spec"，与实证库交叉对照）
- claim 修改走产品 RFC + 法务审；不允许产品 sprint 内"顺手改个数字"
- A/B 测试中的 claim 变体必须每个变体都有 SourceID

### 4.4 客户合同 / 商务材料

- 客户合同条款级别的承诺（"性能不低于 X" / "支持 X 功能" / "Y 工作日内交付"）按 **L4** 处理
- 任何条款级别的承诺需法务 + 产品 + 销售三签
- 商务提案中的 ROI 测算必须基于实证库 SourceID（不允许销售自创数字）
- RFP 响应中的能力声明须走"RFP 响应模板 → SourceID 校验 → 法务审"流程

### 4.5 PR / 媒体材料

- 接受媒体采访前向媒体提供"可引用 claim 清单"（带 SourceID 但脱敏元数据）
- 媒体引用我方数据如有偏差：要求媒体修订；超 **3 工作日**未修订**我方主动发"勘误声明"**——不能依赖媒体后续修订兜底
- 行业排行 / 评奖如要使用我方提供数据，按 §1.2 表格审批
- 同行 / 第三方报告引用我方数据：被动监测，发现偏差按上述同流程

### 4.6 学术合作发布

- 与 KOL / PI / 医院联合发表的研究成果：发布前我方法务 + 学术 + 对方法务 / 学术 + 我方市场全员审
- 我方对该研究的转述（marketing 化简化）须经原作者 PI 书面确认——不允许 marketing 自行简化引用
- 学术合作的"商业化转述"另立 SourceID（与原研究 SourceID 双向关联）
- **合同模板化**（v0.2 修订 · 2026-05-07）：PI 书面确认流程通过**合同模板**预定义——所有学术合作合同附件必含「商业化转述同意条款」，包括：转述用途范围（marketing / 销售 / PR / 投资者关系）/ 修订与撤回机制 / PI 离职 / 反目时的兜底条款 / 转述署名规则。模板由法务起草并维护版本，每年评估一次

## 5. 跨场景一致性

### 5.1 同一 claim 在不同物料的表述一致性

- 同一 claim 在白皮书 / 网站 / UI / 销售 PPT / 客户合同里表述应**一致**——不允许营销端"放大"工程端的保守表述
- 营销与工程对同一数字的表述如有差异，以最保守表述为准；如有理由放大需重新走 §1.2 审批
- 中英对照物料：同一 claim 的中英文版本须语义一致；不允许"中文夸张 + 英文克制"或反之

### 5.2 受众分层的实证级别（最低底线）

> **v0.2 修订 · 2026-05-07**：原 §5.2"受众分层的实证适配"表格已删除——避免与 03e §2.2 引用密度分层重复。本文档 **§1.2 实证矩阵管最低实证级别（合规底线）**；**引用源类型偏好** 回 03e §2.2 处理。

**铁律**：

- §1.2 实证矩阵在所有受众层级**无差别生效**——临床效果 claim 在 T1 三甲与 T4 县域物料中都需 L3+L2，不因受众下沉而降级
- **降低受众的术语密度 ≠ 降低实证级别**——T4 县域物料的引用源数量可少（按 03e §2.2 偏好），但每条仍需 SourceID + 满足 §1.2 最低级别
- 引用源**偏好**（SCI / 国家指南 / 中文综述 / 客户 case 等）按 03e §2.2 受众分层选择

### 5.3 危机场景中的实证留痕

03a §5.1（撤稿）/ §5.2（监管公告）/ §5.6（自家 bug）触发时——所有"危机响应稿件"本身也需 SourceID 化：

- 危机声明的事实陈述部分必须有 SourceID
- 不允许危机响应稿件"绕过"实证流程加速
- 24h 内异步双签场景下：法务在事后 12h 内补审记录

## 6. 工具与基础设施

### 6.1 实证库工具选型

> **v0.2 锁定**（2026-05-07 Sponsor + 法务一致 / 待 Sponsor + 法务 + 工程三方决议会议选定具体方案）：**开源自建**。
>
> v0.2 落地后 30 天内启动工具选型 RFC（Sponsor 在 v0.2 落地清单中加硬节点）。

**候选**（开源自建）：

| 候选 | 优 | 劣 |
|---|---|---|
| **NocoDB**（基于 PostgreSQL，UI 类 Airtable）| 开源 / 上手快 / 数据可迁移 | 需自托管运维 |
| **Baserow**（开源 Airtable 替代）| 开源 / 协作友好 | 大规模检索性能待验证 |
| **自建 Postgres + FastAPI + 简易前端** | 完全可控 / 可与产品 UI claim spec 直连 / 撤稿监测 + 自动校验脚本灵活 | 工程成本 / 维护成本 |

**已排除**：

- ~~Notion 数据库~~：闭源 SaaS / 数据出境合规风险（详见 03e §4.4）/ 大规模检索性能差
- ~~Confluence + Jira 联动~~：闭源 / 需 license / 数据出境合规风险

**工具能力清单**（v0.2 实证库选型 RFC 必须满足）：

- 一键注册（草稿 → 提交 → 审批工作流）
- 自动反向追踪（一键查看哪些 claim 引用此 SourceID + 撤稿时影响范围分析）
- 自动复审提醒（按 §2.4 时效配置季度 / 年度）
- 撤稿监测自动化（PubMed RSS / Retraction Watch RSS 自动入库 + 自动 @ 责任人；NMPA / 国家自然基金委人工每月查）
- 文案 → SourceID 自动校验（发布前一键校验，缺漏标红 + 阻塞发布）
- 多角色协作 UI（市场起草 / 法务审 / 学术审 / 产品引用 / 管理员维护）
- Audit Trail 全留痕（不允许静默删除；只能"标记 revoked + 保留历史"）
- 数据本地化部署（不出境，详见 03e §4.4）

### 6.2 撤稿监测自动化

- PubMed E-utilities API：批量查询 PMID 状态变化
- Retraction Watch RSS：订阅
- 实证库后台 cronjob：每月自动跑一次扫描，新增 revoked 状态自动 @ 责任人（不依赖任何外部 SaaS webhook）
- 国家级（NMPA / 自然基金委）撤稿无标准 API，需人工每月查

### 6.3 文案 → SourceID 自动校验

- 营销 / 投资者材料：上线前文案中的数字 / 引用对照实证库 approved 列表自动校验
- 产品 UI claim spec：与实证库双向同步——产品改了 UI claim 自动触发 SourceID 复核
- 媒体投放素材：发布前同样校验

### 6.4 留痕（Audit Trail）

每条 SourceID 的全生命周期事件留痕：

- 录入 / 审批 / 复审 / 撤销 / 修订
- 关联 claim 的新增 / 移除
- 谁在何时做了什么操作
- 不允许静默删除事件——只能"标记 revoked + 保留历史"

## 9. 自检清单

每条对外声明发布前对照下列 14 条：

- [ ] **SourceID 注册**：该 claim 是否有 SourceID 进入实证库？
- [ ] **状态 approved**：SourceID 状态是否为 approved（非 pending / revoked）？
- [ ] **实证级别匹配**：实证级别（L1–L4）是否匹配 claim 类型（§1.2）？
- [ ] **时效内**：来源时效性是否在 §2.4 推荐范围内？是否经过最近一次复审？
- [ ] **不在红线列表**：来源是否落在 §2.3 不允许列表（撤稿 / Predatory / AI 综述未核验 / 客户口头同意 / 兄弟业务挪用 / 等）？
- [ ] **数据格式**：是否带 (n=X, 来源, 年份)？数字格式符合 03 §7.1？
- [ ] **受众分层适配**：实证级别是否满足 §1.2 矩阵最低底线（无差别要求）？引用源类型偏好是否匹配目标受众层（参 03e §2.2）？
- [ ] **跨语言原文保留**：跨语言引用是否保留原文？译稿是否标注作者翻译？
- [ ] **客户 case 书面同意**：客户 case 是否有书面同意书 + 仍在有效期 + 客户未撤回？
- [ ] **审批 trail 完整**：审批人 / 时间 / 签名留痕是否完整？是否按 §7 审批权矩阵签齐？
- [ ] **修改重审**：自上次审批以来是否有修改？修改后是否重新走审？
- [ ] **跨场景一致**：同一 claim 在白皮书 / UI / 合同 / PPT 中表述是否一致？
- [ ] **媒体校对义务**：媒体引用我方数据是否经我方提供 + 校对？偏差是否触发勘误流程？
- [ ] **PITL 站位**（继承 03 §3.3）：临床效果 claim 是否标注医生角色与责任边界？
